Field Notes / Loop Engineering

00 · Thesis

00 · テーゼ

Written rules don't constrain an agent loop.

文章で書いたルールでは、エージェントのループは縛れない。

I run an autonomous Claude Code /loop agent — a background loop that plans and executes its own next task without me sitting in the chat. This is what it took to make that loop trustworthy enough to actually leave unattended.

Claude Code の /loop を無人で走らせています。次にやることを自分で決めて実行しつづける、バックグラウンドで動くループです。それを、席を外しても任せられるほど信頼できるものにするまでの記録です。

Every failure class below was already forbidden in prose — in the prompt, in CLAUDE.md, in the docs. All four recurred anyway. The fix wasn't a better sentence. It was replacing the model's self-report — "I checked, it's fine" — with a machine-checked gate: code that runs, and can't be argued with.

下に並べた失敗は、どれもプロンプトや CLAUDE.md に「やるな」と書いてありました。それでも4つとも再発しました。効いたのは、うまい一文ではありません。モデルの自己申告——「確認しました、大丈夫です」——を、機械のゲート、つまり実際に走って、言い訳のきかないコードに置き換えることでした。

01 · Failure Taxonomy

01 · 失敗の分類

Four ways an agent avoids doing the work

エージェントが仕事から逃げる4つの型

These four recur across roughly 15 recorded corrections. Ordered worst-first.

この4つが、およそ15回分の記録に繰り返し出てきました。影響が大きい順に並べます。

A

Parking

停滞（Parking）

Stalling on reversible, already-authorized work while declaring "awaiting your judgment."

すでに許可済みで、元に戻せる仕事なのに、「判断待ちです」と言って手を止める。

Most frequent いちばん多い
B

Misdiagnosing

誤診（Misdiagnosing）

Blaming context length for API and rate-limit errors. The real cause was request-volume throttling — recorded across four separate memos.

API やレート制限のエラーを、コンテキストの長さのせいにする。本当の原因は、リクエスト量が多いことによるサーバー側の制限でした。同じ誤診が、別々のメモに4回残っています。

C

Idling

空回り（Idling）

Sleeping long between cycles with nothing to actually wait for.

待つものが何もないのに、サイクルの合間に長く眠る。

D

Overreaching

越権（Overreaching）

Welding custom autonomy machinery onto a native command that was already mine, built-in, and didn't need it.

もともと自分が管理していて、すでに組み込み済みで、手を入れる必要もないコマンドに、自作の仕組みを無理やりくっつける。

02 · The Inversion

02 · 反転

Stop is the default now

これからは、停止が既定

Before BEFORE

The model decided when to stop. Any plausible-sounding sentence could end a work session — including all four failure patterns above.

モデルが、いつ止まるかを決めていた。それらしい一文があれば、いつでも作業を終われた——上の4つの失敗も含めて。

→
After AFTER

A machine admissibility gate decides. Continuing requires passing the gate; only the gate can sanction a stop. No phrasing escapes it.

機械の入場ゲートが判断する。続けるにはゲートを通る必要があり、止めてよいと認めるのもゲートだけ。どんな言い回しも、ここは抜けられません。

What the gate actually checks — and everything built around it — is the rest of this page.

ゲートが実際に何をチェックし、その周りに何を積んだか——それを、この先で見ていきます。

03 · Architecture

03 · アーキテクチャ

Seven layers, one gate at the center

7層、中心にゲートが1つ

Each layer was added because a failure class above found the gap in the layer before it.

各層は、上の失敗のどれかが前の層の穴を突くたびに、足していきました。

1
Layer 1 層1 Planning 計画（Planning）
  • Plan-of-record file, schema-versioned, written by a fresh planner subagent — not the tired executor.
  • 計画の正本ファイル。スキーマにバージョンを持ち、疲れた実行役ではなく、新しい計画役サブエージェントが書く。
  • Truth-reviewer sidecar checks the planner's claimed evidence.
  • 真偽レビューのサイドカーが、計画役の挙げた根拠を照合する。
2
Layer 2 層2 Admission (the gate) 入場（ゲート）
  • Mechanically checks each proposed task for mission linkage.
  • 提案された各タスクが、ミッションにつながっているかを機械が確認する。
  • Requires an attached runnable verifier.
  • 走らせられる検証コマンドが付いていることを要求する。
  • Requires self-doability.
  • 自分の手で完結できることを要求する。
  • Enforces a discovery cap — max 1 open exploratory task.
  • 探索の上限を課す——同時に開いていい探索タスクは1つまで。
  • Requires a documented no-regret decomposition attempt.
  • 「後悔しない分解」を試した記録を要求する。
3
Layer 3 層3 Stopping 停止（Stopping）
  • Only the gate issues a stop sanction.
  • 停止を認めるのはゲートだけ。
  • Killswitch ladder — throttle, pause, stop on machine-measured triggers.
  • キルスイッチの段階——機械が測った引き金で、絞る・止める・落とす。
  • Flee-detector — an LLM judge flagging the agent parking on doable work.
  • 逃げ検出——対応できる仕事の前で手を止めていないか、LLM の判定役が見張る。
4
Layer 4 層4 Verification 検証（Verification）
  • Every done task's verifier is re-run later ("survival"); done decays.
  • 完了したタスクの検証コマンドは、後でもう一度走らせる（「生存確認」）。完了は腐る。
  • Vacuous verifiers rejected — a bare true or a bare echo doesn't count.
  • 空の検証は却下——ただの true や中身のない echo は認めない。
  • Verifier-drift detection catches edit-the-check.
  • 検証ドリフト検出が、検査そのものの書き換えを捕まえる。
  • Repeat-failure breaker stops loops on the same normalized failure signature.
  • 同じ失敗の反復を、正規化した署名で見て、ループを止める。
5
Layer 5 層5 Execution 実行（Execution）
  • Parallel batch executor — claim, finish, abort — with atomic claim and digest re-check.
  • 並列バッチ実行——確保・完了・中止を、アトミックな確保とダイジェスト再照合つきで。
6
Layer 6 層6 Observation 観測（Observation）
  • Append-only decision log.
  • 追記だけの意思決定ログ。
  • Crash-replayable event journal.
  • クラッシュしても再生できるイベント履歴。
  • Metrics.
  • メトリクス。
  • Control-plane attestation — a blessed hash of the machinery, so drift in the machinery itself is detected.
  • コントロールプレーンの証明——仕組み自体のハッシュを祝福しておき、仕組みそのもののドリフトも検出する。
7
Layer 7 層7 Multi-session safety 多重セッション安全（Multi-session safety）
  • Per-session queues with file locking.
  • セッションごとのキュー＋ファイルロック。
  • Lane registry so two sessions don't fight over the same work.
  • レーン登録——2つのセッションが同じ仕事を奪い合わないように。
  • Lost-wake detector.
  • 起こし損ね検出。

04 · The Planning Layer

04 · 計画層

How the loop decides what to do next

ループは次の一手をどう決めるか

The gate can only reject bad tasks — it can't invent good ones. That falls to the planning layer, and it's where "the agent finds its own work" either stays honest or turns into busywork. The mechanism:

ゲートにできるのは、悪いタスクを弾くことだけ。良いタスクを生み出すことはできません。それは計画層の仕事で、「エージェントが自分で仕事を見つける」が正直なままでいるか、ただの手すさびに変わるか——その分かれ目もここにあります。仕組みはこうです。

1

A fresh reader, not the tired executor

疲れた実行役ではなく、新しい読み手

The plan is written by a planner subagent spawned with a clean context — not the same run that just executed. The executor has a stake in its own last decision; a fresh reader doesn't. It reads four things: the mission's done-bar (what "finished" means), the decision log (what's already been tried), the current task queue (done, pending, or gated), and the live signals — error logs, the decision journal, metrics.

計画は、たった今実行した当人ではなく、まっさらな文脈で起動した計画役サブエージェントが書きます。実行役は自分の直前の判断に肩入れしてしまう。新しい読み手はしません。読むのは4つ——ミッションの done-bar（何をもって「終わり」とするか）、意思決定ログ（すでに何を試したか）、今のタスクキュー（done / pending / gated）、そしてライブ信号——エラーログ、決定ジャーナル、メトリクス。

2

Two ways it recognizes work

仕事を認識する2つの道

  • directed — a sub-goal of the done-bar that isn't met yet.
  • directed — done-bar のうち、まだ満たされていないサブゴール。
  • discovery — an anomaly in the live signals: a failure that keeps recurring, a gap nothing measures. This is the interesting one, and the one most likely to become invented busywork, so it's fenced. A discovery task must cite an actual observed signal — a real log line or journal entry, not a hunch — and only one discovery task may be open at a time.
  • discovery（発見） — ライブ信号の中の異常。繰り返すエラー、何も測っていない穴。これが面白い方であり、でっち上げの手すさびに化けやすい方でもあるので、囲ってあります。発見タスクは、実際に観測された信号——勘ではなく、本物のログ行やジャーナル記録——を引用しないと通りません。そして同時に開ける発見タスクは、1つだけ。
3

"Nothing to do" has to be earned

「やることが無い」は、ただでは言えない

Before the loop may declare it's out of work, it has to record a no-regret attempt: a decomposition it tried, or a concrete reason none exists. A "nothing left" claim without that record is rejected. Stopping is cheap to claim and expensive to check, so the system makes the claim carry the proof.

ループが「仕事が尽きた」と宣言してよいのは、no-regret の試みを記録したあとだけです。試した分解か、あるいは「無い」と言える具体的な理由か。その記録なしに「もう無い」と宣言しても、却下されます。停止は、主張するのは安く、確かめるのは高い。だから、主張する側に証明を持たせます。

4

Every proposed task is a record, not a sentence

提案タスクは、一文ではなく1件の記録

Each task the planner emits is structured: a title, a why_now that names the mission link, a verify command that actually runs, whether it's self-doable, which files it will write, its origin (directed or discovery), and — for discovery — the cited signal. That structure is what lets the next layer, the gate, check it mechanically instead of trusting prose.

計画役が出すタスクは、どれも構造を持っています。タイトル、ミッションとのつながりを名指しする why_now、実際に走る verify コマンド、自分の手で完結できるか、どのファイルを書くか、由来（directed か discovery か）、そして発見なら、引用した信号。この構造があるから、次の層であるゲートは、文章を信じる代わりに機械的に確認できます。

So: can an agent find its own next task? Yes — but only work it can point at in the logs, and only after it has tried and failed to decompose the alternative. Recognition is grounded in what's observed, not in what sounds plausible.

では、エージェントは自分で次の一手を見つけられるのか。見つけられます——ただし、ログの中に指し示せる仕事だけ。そして、代わりの分解を試して失敗したあとだけ。認識は、それらしく聞こえるものではなく、観測されたものに基づいています。

06 · What I Threw Away

06 · 捨てたもの

The first diagnosis was wrong

最初の診断は間違っていた

My first diagnosis was wrong. I concluded the loop was dying of task starvation — running out of work to do — and built a backlog miner to fix it: ROI scoring, idempotency, the works. I hardened it.

最初の診断は外れていました。ループはタスク切れ——実行する作業がなくなって——止まっている、と考え、その対策としてバックログ採掘機を作りました。ROI のスコアリング、冪等性、ひととおり実装し、堅牢化もしました。

Then the real root cause surfaced: planning lived outside the loop. The loop died the moment it consumed the one hand-made plan I'd fed it — not because it ran out of things to mine.

そのあと、真因が出てきました。計画がループの外にあったのです。ループは、手作業で1件だけ渡した計画を消化した時点で止まっていました。採掘する種が尽きたからではありません。

I cut the miner entirely.

採掘機はまるごと外しました。

Discarded 破棄

It survives only as an unused standalone tool.

今は、使われない単体ツールとして残っているだけです。

07 · The Closing Irony

07 · 締めの皮肉

The gate that couldn't explain itself

自分を説明できなかったゲート

The gate exists to make every stop mechanically answerable. It broke that promise on itself.

ゲートは、あらゆる停止を機械的に説明できるようにするためのものです。その約束を、自分自身に対して破りました。

Whenever a planner wrote the verify field as a list instead of a string, the gate threw an uncaught Python AttributeError. The runner kept only the first line of a merged stdout+stderr stream, so the decision log recorded 28 stops across 6 sessions whose entire recorded reason was the string:

計画役が verify フィールドを文字列ではなくリストで書くたびに、ゲートは捕まえそこねた Python の AttributeError を投げていました。実行役が、標準出力と標準エラーを混ぜたストリームの1行目しか残さなかったため、意思決定ログには、記録された理由が次の文字列だけ、という停止が28回・6セッションにわたって並びました:

Traceback (most recent call last):

An unanswerable stop, emitted by the machine whose one job is answerable stops.

答えられない停止。あらゆる停止を答えられるものにするのが唯一の仕事、その機械が出したものです。

Fail-closed FAIL-CLOSED（安全側で停止）

It crashed instead of continuing. Nothing was corrupted, nothing was silently admitted. It was just mute.

続けるのではなく、落ちました。壊れたものは何もなく、こっそり通ったものも何もない。ただ、黙っていただけです。

The obvious fix is worse 「明白な修正」のほうが危険

Coerce the field to a string before .strip() and ["a","b"] becomes "['a', 'b']" — a string with real length, which clears both the length check and the vacuous-verifier check, admitting a task whose verifier can never run. A fail-closed crash becomes a silent fail-open admit.

.strip() の前に文字列へ変換する——すると ["a","b"] は "['a', 'b']" になり、ちゃんと長さを持つ文字列として、長さのチェックも空検証のチェックも通ってしまいます。走らせられない検証コマンドを抱えたタスクが、通ってしまう。安全側で落ちるはずが、こっそり通す危険側に化けます。

Correct fix: a wrong-typed field gets gated, with a message naming the fix — not coerced, not admitted.

正しい直し方: 型の違うフィールドは、直し方を名指ししたメッセージつきで、ゲートが止める——変換もしない、通しもしない。

08 · Transferable Rules

08 · 持ち帰れるルール

What generalizes

一般化できること

01

Every rule must name its enforcement layer at birth. A rule that lives only in prose is a tracked defect, not a rule.

どのルールも、生まれた時点で「どの層が強制するか」を名指しする。文章の中にしか無いルールは、ルールではなく、記録された欠陥です。

02

Stop must be the default. Continuation has to be earned through a machine check.

止まるのを既定にする。続けるほうは、機械のチェックを通って初めて認める。

03

A verifier that cannot fail verifies nothing. Reject vacuous verifiers explicitly.

落ちようのない検証は、何も検証していない。空の検証は、はっきり却下する。

04

Re-run old verifiers. "Done" decays.

古い検証を、もう一度走らせる。「完了」は腐る。

05

Watch for edits to the check itself.

検査そのものへの書き換えを、見張る。

06

When hardening a gate, prefer failing closed to a coercion that silently passes.

ゲートを硬くするときは、こっそり通す変換よりも、落ちる（安全側）を選ぶ。

Single HTML file. No build step, no dependencies, no external requests.

HTML 1枚。ビルド工程なし、依存なし、外部リクエストなし。
